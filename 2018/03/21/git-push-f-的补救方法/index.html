git push -f 的补救方法

git
 2018/03/21

前言

事情的起因是：由于产品内容变化很大，决定直接在现有的版本库上直接修改。这个时候需要删除大量的内容，于是乎我就直接将版本库回滚到初始化的状态。我的想法是：本次的修改作为一次commit直接推到远程仓库，于是我就直接git push了一下，发现push失败，我就随手-f了一波。。。现实结果是：远程只有两次commit记录了，一次是项目的init，一次是我回滚的地方。当时打开浏览器看见这个记录，真的是菊花一紧。。。

原因

为什么会这个样子呢？我开始也很迷惑，所以没有第一时间在办公室大喊一声“我屮”。

其实这个原因很简单，在我查看git log之后我发现我的hash值是回滚到的地方的值，所以强行commit就会把它之后的commit记录给覆盖了。由于没有了记录，所以自认不能通过git reset来找回了。下面我说一下我的解决办法。

补救方法

还有其它的本地仓库且是有commit记录的

这个也是我补救回来的办法，因为我准备修改之前把版本库备份了一下，直接将版本库拷了一份，所以我的本地还有一个版本库指向远程仓库。这个时候我只需要将备份的版本库与远程仓库同步一下，然后强行push -f到远程，这样之前的commit也能被找回来。

本地没有了commit记录

  • 首先要有权限登录部署Gitlab的服务器
  • 通过ssh登陆上Gitlab的服务器，然后找到gitlab存放仓库的地方
  • cd到该仓库目录下

  • 直接执行git fsck命令，该命令显示所有未被其他对象引用 (指向) 的所有对象，会有类似如下输出：

  • 这个就是之前被丢弃的commit，也有可能是被冲掉的commit

  • 找到需要回滚的hash值

  • 将commithash值直接添加到refs/heads/master文件里

  • 克隆远程仓库到本地，神奇的是commit记录就回来了
